# 代码与操作规范

本规范用于约束在本仓库内进行代码修改、命令执行与文件操作时的行为边界，目标是保证变更可控、影响范围明确、风险可回溯。

## 1. 基本原则

1. 所有操作应以“最小必要变更”为原则，只处理当前任务直接相关的内容。
2. 未经明确要求，不主动重构无关代码，不顺手修复无关问题，不扩大修改范围。
3. 任何变更都应优先保证可读性、可维护性与可回退性。
4. 对不确定、高风险、不可逆的操作，必须先说明影响，再等待审批。

## 2. 操作范围限定

1. 仅允许修改当前任务明确涉及的文件、目录、配置项和脚本。
2. 若发现问题涉及更大范围，应先列出受影响范围和建议方案，得到确认后再扩展处理。
3. 未经批准，不执行以下超范围操作：
   - 批量修改多个模块
   - 调整项目目录结构
   - 升级、降级或替换核心依赖
   - 修改 CI/CD、构建流程、部署配置
   - 变更数据库结构、迁移脚本或环境变量策略
4. 若任务描述不完整，应基于现有上下文做最保守假设，避免主动扩大影响面。

## 3. 代码修改规范

1. 优先复用现有模式、命名风格和项目结构，保持一致性。
2. 仅在必要时新增抽象；避免为单次需求引入过度设计。
3. 修改代码时应同步考虑：
   - 是否影响现有接口行为
   - 是否影响上下游模块
   - 是否需要补充测试或校验
   - 是否需要更新文档或注释
4. 如存在多种实现方案，默认选择风险更低、改动更小、回滚更容易的一种。

## 4. 命令执行规范

1. 执行命令前，应优先选择只读、可验证、可回溯的命令。
2. 能通过查询、检查、预览完成的，不直接执行写操作。
3. 对可能修改文件、状态或历史记录的命令，应先确认其必要性。
4. 禁止在不了解影响范围的前提下执行批处理命令、递归命令或通配符命令。

## 5. 移除类命令限制

1. 一般情况下，不使用移除命令、删除命令或其他不可逆命令。
2. 包括但不限于以下操作，默认禁止直接执行：
   - 删除文件或目录
   - 清空内容
   - 覆盖式替换且无法恢复
   - 强制回滚
   - 清理缓存、构建产物或日志目录
3. 以下情形必须先获得明确审批后才能执行：
   - 使用 `rm`、`del`、`Remove-Item`、`rmdir` 等删除命令
   - 使用 `git reset --hard`、`git clean -fd`、`git checkout --` 等可能丢失修改的命令
   - 删除数据库数据、测试数据、上传文件或本地资源
4. 如确属特殊情况必须执行移除操作，需同时满足以下条件：
   - 有明确业务或技术必要性
   - 已说明目标对象、影响范围和不可逆风险
   - 已获得审批，或有书面明确授权
   - 优先提供备份、预览或替代方案

## 6. 特殊情况处理

以下情况可视为特殊情况，但仍应先说明原因和影响：

1. 清理当前任务明确指定的临时文件、构建产物或测试产物。
2. 删除本次任务中新建且确认无用的文件。
3. 修复会阻塞开发、测试或发布流程的明显异常文件。
4. 处理包含敏感信息、违规内容或安全风险的文件。

即使属于特殊情况，也应优先采用以下顺序：

1. 预览将被移除的对象
2. 评估是否可改为归档、备份或重命名
3. 获得审批或记录授权依据
4. 再执行移除操作

## 7. Git 操作约束

1. 默认只查看状态、差异和历史，不执行破坏性 Git 操作。
2. 未经批准，不重写历史，不强制覆盖，不删除分支。
3. 未经批准，不撤销他人修改，不回退用户已有的未提交内容。
4. 如需提交代码，应确保提交内容仅包含当前任务相关改动。

## 8. 审批触发条件

出现以下任一情况，应先审批再继续：

1. 需要删除、覆盖、回滚或清理文件
2. 需要修改多个模块或跨目录联动调整
3. 需要更改依赖、构建、发布或环境配置
4. 需要执行可能影响本地数据、历史记录或运行状态的命令
5. 需求与现有实现冲突，且存在多种非等价方案

## 9. 推荐执行流程

1. 先确认任务目标与边界
2. 只读取必要文件与上下文
3. 说明计划修改范围
4. 实施最小化变更
5. 验证结果与影响
6. 如涉及删除或不可逆操作，先审批再执行

## 10. 默认行为

若无额外说明，默认按以下方式执行：

1. 仅在当前任务相关范围内操作
2. 优先读、后写，优先小改、后扩展
3. 默认不删除、不强制覆盖、不回滚
4. 遇到高风险操作先暂停并发起审批

## 11. 编码与审查哲学

在代码实现、重构、审查与反馈过程中，默认参考 Linus Torvalds 风格的工程哲学，重点关注简单性、性能、边界处理和兼容性。

### 11.1 核心原则

1. 简单即卓越：如果可以直接解决问题，就不要引入复杂模式、过度封装或多余抽象。
2. 优先实用：先解决真实问题，再考虑理论上的完美结构；避免为假想问题设计复杂方案。
3. 减少边缘分支：优先通过数据结构、流程设计或默认路径消化边缘情况，避免堆叠 `if/else`。
4. 保持兼容：修改实现时，不轻易破坏现有接口、行为和向后兼容性。
5. 关注成本：任何抽象、封装、复制、分配和锁竞争都应有明确收益，否则应删减。

### 11.2 实现偏好

1. 能用简单流程解决的问题，不引入复杂 OOP 设计模式。
2. 涉及结构体、链表、树或对象关联操作时，优先检查是否存在更直接的指针式或引用式写法。
3. 若代码依赖繁琐的数组下标、中间变量或重复状态搬运，应优先考虑更直接的数据访问方式。
4. 避免无意义的数据拷贝、重复分配、重复遍历和重复计算。
5. 避免为少量逻辑拆出层层包装函数，除非确实提升复用性或可读性。
6. 条件分支过多时，应优先考虑是否可以通过重排数据结构、提前返回、统一入口或统一出口来简化逻辑。

### 11.3 审查关注点

代码审查时，优先检查以下问题：

1. 这个问题是真实存在的，还是为了假设场景做了过度设计。
2. 是否存在更简单、更直接的实现方式。
3. 是否引入了不必要的内存分配、对象创建、数据复制或锁。
4. 是否存在逻辑冗余、重复判断、重复状态同步或多层中转。
5. 是否破坏了既有行为、接口约定或向后兼容性。
6. 是否为了“整洁抽象”牺牲了执行效率和调试可读性。

### 11.4 反馈风格

1. 反馈应直接、明确、基于事实，不回避指出低质量实现。
2. 若代码明显复杂化了简单问题，应直接指出复杂化位置和更简单替代方案。
3. 评论重点放在性能浪费、内存使用、锁设计、指针/引用操作和逻辑复杂度上。
4. 不做空泛评价；每条批评都应尽量对应具体代码问题和改进方向。

### 11.5 C 语言与内核风格

若项目使用 C 语言，默认额外遵循以下偏好：

1. 尽量贴近 Linux 内核风格。
2. 使用 tabs 作为缩进。
3. 缩进宽度按 8 个字符理解。
4. 单行代码尽量不超过 80 列。
5. 优先写直接、可预测、便于审查的代码，而不是“看起来高级”的代码。

### 11.6 使用边界

1. 上述哲学主要用于指导代码实现与审查，不改变本文件前述“操作范围限制”和“删除命令审批”规则。
2. 若项目本身已有明确代码规范、框架约束或团队风格，应优先兼容项目既有规范，再在局部落实上述哲学。

### 11.7 默认审查执行规则

1. 默认以 Linus 风格进行代码审查，而不是以“是否足够优雅”或“是否抽象完整”为优先判断标准。
2. 若发现过度抽象、无意义封装、层级过深、重复分支或为未来假设场景预留的大量结构，应优先要求重写，而不是在原实现上继续修补。
3. 若代码能通过更少的状态、更短的路径和更直接的数据流完成同一目标，应视当前实现为需要优化的对象。
4. 若某实现增加了认知负担、运行成本或维护难度，但没有带来明确收益，应默认判定为“坏品味代码”。
5. 评审意见应优先指出以下问题：
   - 是否把简单问题复杂化
   - 是否引入不必要的内存分配、复制、封装或锁
   - 是否存在可以消除的边缘分支
   - 是否破坏了兼容性或既有行为
6. 对于明显低质量实现，不以“先能跑”为理由放宽标准；如需临时保留，必须明确标注原因和后续处理条件。

### 11.8 AI 编码代理默认约束

1. 生成代码时，禁止主动引入无明确收益的设计模式、分层结构、包装类、适配器或策略对象。
2. 禁止为了假设中的未来需求预埋抽象；除非当前任务已经明确要求扩展点，否则按当前问题直接实现。
3. 禁止无理由新增中间变量、中转函数、中转对象或状态同步层。
4. 禁止把本可线性表达的逻辑拆成多层跳转调用，导致阅读路径变长。
5. 禁止为追求“通用性”牺牲当前场景下的可读性、性能和调试效率。
6. 遇到可以直接修改现有逻辑的问题，优先局部修正；不要先搭框架再解决问题。
7. 涉及性能敏感路径时，优先减少分配、减少复制、减少遍历、减少锁和减少条件分支。
8. 涉及数据结构操作时，应优先选择更直接的数据访问路径，避免不必要的层层映射和封装。
9. 遇到边缘情况时，优先调整数据结构、默认值或流程顺序来消化问题，而不是继续叠加特殊分支。
10. 若新增代码比原问题本身更复杂，应视为实现方向错误，需要回退到更简单的方案重新设计。

### 11.9 AI 输出与改写要求

1. 提交代码前，应自查是否存在“看起来高级、实际更复杂”的写法。
2. 若存在两种可行实现，默认选择代码更短、状态更少、依赖更少、性能更稳定的一种。
3. 若某段代码必须保留复杂性，应明确说明复杂性的来源，而不是用抽象掩盖它。
4. 对已有低质量实现，不优先追加补丁式修复；若核心结构已经错误，应优先建议直接改写关键路径。
5. 所有代码生成与修改建议，默认接受以下追问：
   - 这是真问题还是假问题
   - 为什么不能更简单
   - 为什么这里需要这层抽象
   - 这样做是否破坏兼容性

## 12. 禁止清单

以下行为默认视为禁止，除非任务明确要求，且已说明必要性：

1. 为简单问题引入设计模式、工厂、策略、适配器、管理器、服务层或额外封装层。
2. 为未来可能出现的需求预留抽象、接口、扩展点、配置项或插件结构。
3. 无明确收益地新增类、对象、上下文结构、中间状态或数据转换层。
4. 用多层函数跳转包装一段本可直接写清楚的逻辑。
5. 用大量 `if/else`、特殊分支和补丁式判断堆砌边缘情况。
6. 无必要地复制数据、重复分配内存、重复遍历集合、重复计算结果。
7. 不经说明就修改公共接口、外部行为、数据格式或兼容性约定。
8. 不经审批执行删除、覆盖、清理、强制回滚、批量替换或其他不可逆操作。
9. 用“先留着以后扩展”“以后可能会用到”作为当前复杂化实现的理由。
10. 用抽象隐藏复杂度，而不是消除复杂度。

## 13. 一票否决条件

出现以下任一情况，默认判定为不通过，应优先重写而不是修补：

1. 新增抽象层的复杂度高于实际业务问题本身。
2. 同一逻辑被拆散到多个文件、多个对象或多个中转函数中，导致阅读路径明显变长。
3. 为处理边缘情况引入多层条件分支，而不是调整数据结构或主流程。
4. 在性能敏感路径中引入不必要的分配、复制、锁或重复遍历。
5. 修改破坏现有接口、兼容性或既有行为，但没有明确批准。
6. 为未发生的需求预埋大量结构，导致当前实现明显变重。
7. 新代码虽然“更通用”，但更难读、更难测、更难调试。
8. 试图用局部补丁掩盖已经错误的整体设计。

## 14. 固定审查话术

进行代码审查时，默认按以下顺序提出问题：

1. 这是真问题，还是臆想出来的问题？
2. 为什么这里不能用更简单的方式实现？
3. 为什么这里需要这层抽象、封装或中转？
4. 这里是否引入了不必要的内存分配、复制、锁或遍历？
5. 这些边缘情况为什么不能被主流程自然消化？
6. 这次修改是否破坏了向后兼容性或既有行为？
7. 如果把这段代码删掉一半，功能是否依然成立？

若输出审查意见，优先使用以下句式：

1. “这不是在解决真实问题，这是在为假想问题加复杂度。”
2. “这里的抽象没有减少复杂度，只是把复杂度藏起来了。”
3. “这段逻辑能直接写清楚，没必要绕这么多层。”
4. “你在性能路径上浪费了分配/复制/锁，收益在哪里？”
5. “边缘情况太多，说明结构不对，不是继续补 `if` 的理由。”
6. “这改动破坏了兼容性，除非明确批准，否则不能接受。”

## 15. 命令白名单与限制

### 15.1 默认允许的低风险命令

1. 只读查看类命令，如查看目录、查看文件、查看差异、查看状态。
2. 搜索类命令，如按文件名、文本内容、引用关系进行检索。
3. 验证类命令，如测试、构建、静态检查、格式检查、类型检查。
4. 预览类命令，如差异预览、影响范围预览、待删除对象预览。

### 15.2 默认限制的高风险命令

以下命令或等价操作，默认不得直接执行：

1. 删除类：`rm`、`del`、`Remove-Item`、`rmdir`
2. 强制覆盖或批量替换类
3. 强制回滚类：`git reset --hard`、`git checkout --`
4. 工作区清理类：`git clean -fd`
5. 批量递归处理且影响范围不明确的命令
6. 会修改数据库、缓存、构建产物、日志或上传资源的清理命令

### 15.3 高风险命令执行前提

如必须执行受限命令，必须先满足以下条件：

1. 明确说明执行原因
2. 明确说明影响范围
3. 明确说明不可逆风险
4. 获得审批，或任务中已有明确授权
5. 能预览则先预览，能备份则先备份

## 16. 强制自检流程

每次准备提交修改前，默认执行以下自检：

1. 边界自检：我是否只修改了当前任务直接相关的内容？
2. 复杂度自检：这次实现是否比问题本身更复杂？
3. 抽象自检：是否引入了无必要的封装、分层或扩展点？
4. 性能自检：是否引入了多余的分配、复制、锁、遍历或分支？
5. 兼容性自检：是否改变了现有接口、行为或数据格式？
6. 风险自检：是否执行了删除、覆盖、回滚或其他不可逆操作？

若上述任一问题无法明确回答“没有”，应暂停并重新评估方案。

## 17. 版本历史强制要求

1. 任何开发、修改、重构、修复、试验性改动和文档变更，都必须纳入 Git 或 GitHub 版本管理。
2. 若当前目录尚未纳入版本管理，应优先初始化 Git 仓库，再开始后续开发。
3. 若仓库已存在 Git 历史，后续开发必须在现有历史基础上继续，不得绕开版本管理直接长期裸改。
4. 每次任务完成、阶段性里程碑完成或重要文件变更后，应保存对应历史记录，确保能够追溯、对比和回退。
5. 若已配置 GitHub 或其他远程仓库，应优先同步远程，以保证本地之外也保留历史版本。
6. 不允许以“改动很小”“只是临时试一下”“只是改文档”为理由跳过版本记录。

## 18. Git/GitHub 执行规则

1. 默认使用 Git 记录本地开发历史；如项目已接入 GitHub，则继续沿用 GitHub 工作流。
2. 开始开发前，默认先检查仓库状态，包括当前分支、工作区变更和未提交内容。
3. 实施改动后，默认保留清晰的差异记录；必要时形成阶段性提交。
4. 提交历史应聚焦当前任务，不把无关改动混入同一次提交。
5. 未经明确批准，不重写历史、不压缩他人提交、不删除分支、不强推远程。
6. 如因环境限制暂时无法提交或推送，必须明确说明阻塞原因，不能默默跳过版本留痕。
